# 👍 What You Did Well

## 1. Clear separation of Read vs Write services

You separated:

* **Read Service**
* **Submission Service**

This is good because:

* Reads (problem listing, viewing submissions) are **high traffic**
* Writes (code submission) are **compute heavy**

This separation improves scalability.

✅ Interviewers like seeing **CQRS-style thinking**.

---

## 2. Async submission processing (Queue + Workers)

Your flow:

User → Submission Service → Queue → Worker nodes → Execution

This is **correct architecture**.

Why?

* Code execution is **slow**
* Must not block API

Your design handles:

* burst submissions
* scaling workers
* retries

Very good.

---

## 3. Worker nodes for code execution

You correctly separated:

```
worker nodes
    ↓
compiler containers
```

This is important because:

* Each submission must run in **isolated sandbox**
* Prevent malicious code

Good thinking.

Production systems use:

* Docker
* Firecracker
* gVisor

LeetCode actually uses sandboxing similar to this.

---

## 4. Redis for leaderboard

This is **excellent**.

Leaderboards need:

* **sorted ranking**
* **fast updates**
* **fast reads**

Redis supports:

```
ZADD leaderboard score userId
ZRANGE leaderboard
```

Perfect fit.

---

## 5. Good async result storage

Flow you designed:

```
execute code
↓
store result
↓
read service fetches
```

Correct.

This prevents client waiting.

# ⚠️ Major Improvements

Now the important parts.

These are things **interviewers will expect you to mention**.

---

# 1️⃣ Problem service should be separate

Currently you have:

```
Read Service
```

But ideally we want:

```
Problem Service
Submission Service
User Service
Contest Service
Leaderboard Service
```

Why?

Problems are **admin-managed content**.

They include:

* statement
* constraints
* editorial
* hints
* tags
* difficulty

Better design:

```
User
 ↓
API Gateway
 ↓
Problem Service
 ↓
Problem DB
```

---

# 2️⃣ Missing caching for problems

Problem list is **read extremely frequently**.

Your system currently:

```
Read Service → DynamoDB
```

Instead:

```
Read Service
     ↓
   Redis Cache
     ↓
 DynamoDB
```

Cache:

```
problem list
problem details
tags
difficulty filters
```

Without cache your DB will get overloaded.

---

# 3️⃣ Test cases should NOT be in problem table

You wrote:

```
problem_all_testcases
problem_sample_testcases
```

This is dangerous.

Why?

Testcases may be **large**.

Better design:

```
Problem DB
    ↓
testcase_metadata
```

Actual testcases stored in:

```
S3 / Blob storage
```

Workers fetch:

```
testcase from object storage
```

LeetCode stores testcases separately.

---

# 4️⃣ Code execution sandbox missing

Your diagram shows:

```
worker → compiler
```

But execution must be **secure sandbox**.

Need:

```
worker
   ↓
sandbox container
   ↓
compiler
```

Sandbox must enforce:

* CPU limit
* memory limit
* runtime limit
* system call restrictions

Otherwise malicious code could:

```
fork bomb
rm -rf
network attack
```

---

# 5️⃣ Missing status polling / WebSocket

Currently flow:

```
submission → return id
```

But how does user know result?

Two options:

### Polling

```
GET /submission/{id}
```

or

### Websocket

Push result when finished.

LeetCode uses **polling**.

---

# 6️⃣ Queue type not defined

Queue is critical.

You should specify:

Possible options:

* Kafka
* SQS
* RabbitMQ

Better choice:

```
Kafka
```

Why?

* high throughput
* durable
* replay capability

---

# 7️⃣ Leaderboard architecture needs improvement

You wrote:

```
Redis leaderboard
snapshot stored in contest_table
```

Good idea.

But leaderboard update must handle:

```
fastest submission
penalty time
tie-breaking
```

Better design:

```
Redis Sorted Set

score = problems_solved
tie_breaker = time_penalty
```

---

# 8️⃣ Submission table schema improvement

Your schema:

```
submission_id
user_id
code
language
time
metadata
```

Missing important fields:

```
status (RUNNING / ACCEPTED / TLE)
runtime
memory
testcases_passed
```
